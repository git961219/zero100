## 1. 현재 문제 분석

| 문제 | 원인 | 영향도 |
|------|------|--------|
| 빈 화면이 많다 | 정보 밀도가 낮음, Empty State 미설계 | 높음 - 앱 신뢰감 저하 |
| 전문 측정 앱 느낌 부족 | 계기판 느낌 없음, 평범한 Material 카드 | 높음 - 차별화 실패 |
| 궤적 시각화 없음 | 지도 기능 미구현 | 높음 - 핵심 기능 부재 |
| 결과 화면 밋밋함 | 숫자만 나열, 그래프/지도 없음 | 중간 - 공유 욕구 미자극 |
| 측정 중 정보 부족 | 현재 속도만 표시, 보조정보 없음 | 중간 - 운전 중 활용도 낮음 |

### 벤치마킹 핵심 포인트

| 참고 대상 | 가져올 요소 |
|-----------|-------------|
| Dragy | 결과 화면의 구간 타임 표, 속도-시간 그래프, 리더보드 개념 |
| VBOX Sport | 대형 숫자 중심 측정 화면, 정밀도 강조 UI |
| 티맵 주행기록 | 속도별 그라데이션 궤적, 구간별 마커 |
| 레이싱 게임 HUD | 어두운 배경 + 네온 액센트, 속도계 중심 레이아웃 |

## 2. 디자인 철학 & 원칙

### 핵심 컨셉: "Precision Instrument" (정밀 계측기)

> 이 앱은 장난감이 아니라 정밀한 측정 도구다.
> 모든 디자인 결정은 "계측기"의 신뢰감에서 출발한다.

### 5대 원칙

| 원칙 | 설명 | 적용 예 |
|------|------|---------|
| 1. 어둠 위의 정밀한 빛 | 다크 테마 기본, 핵심 숫자만 밝게 | 배경 #0D0D0D, 속도 숫자 #FFFFFF |
| 2. 한눈에 읽힌다 | 운전 중 0.3초 내 인지 가능 | 속도 숫자 120sp, 단위는 작게 |
| 3. 밀도 있되 혼잡하지 않다 | 빈 공간 = 정보 공간 | 카드 안에 보조 데이터 배치 |
| 4. 데이터가 말한다 | 장식 최소화, 수치 강조 | 불필요한 아이콘/애니메이션 제거 |
| 5. 공유하고 싶은 결과 | 결과 화면 = 스크린샷용 | 로고, 차량명, 날짜 포함 |

```

배경 계층:
  Surface-0 (최하위)    #0A0A0A    순수 다크, 앱 배경

  Text-Primary          #FFFFFF    핵심 숫자 (속도, 시간)

  Text-Tertiary         #666666    비활성, 힌트

  Speed-4    (150-200 km/h)  #FF9800    주황
  Speed-5    (200+ km/h)     #F44336    빨강
```

### 3.2 타이포그래피

| 용도 | 폰트 | 크기 | Weight | 비고 |
|------|------|------|--------|------|
| 메인 속도 | Rajdhani / D-DIN | 120sp | Bold | 모노스페이스 느낌의 기술 폰트 |
| 구간 타임 | Rajdhani | 48sp | SemiBold | |
| 보조 수치 | Rajdhani | 24sp | Medium | |
| 레이블 | Pretendard | 14sp | Regular | 한글 가독성 |
| 단위 (km/h, sec) | Rajdhani | 16sp | Light | 메인 숫자 대비 작게 |
| 버튼 텍스트 | Pretendard | 16sp | SemiBold | |

> 선택 근거: Rajdhani는 무료 Google Font, 계기판/기술 느낌, 숫자 가독성 우수.
> D-DIN은 유료이나 더 세련됨. 택님 선호에 따라 결정.

- 최근 측정 목록으로 활동 히스토리 제공
- START 버튼은 화면 하단에 크게, 항상 접근 가능

### 4.3 측정 중 화면 (Live Measurement)

**목표**: 운전 중 한눈에 확인. 대형 숫자. 불필요한 정보 최소화.

```

|              |    sec     |                        |
|              +------------+                        |
|           (경과 시간, 1/100초 단위)                  |
|                                                    |
+--------------------------------------------------+
|                                                    |
|  +--------------------------------------------+   |
|  |  구간 달성                                    |  |
|  |                                              |  |
|  |   30 km/h    1.82s   [====]                 |  |
|  |   60 km/h    3.14s   [========]             |  |
|  |   80 km/h    4.01s   [===========]          |  |
|  |  100 km/h    ---     [                ]     |  |
|  +--------------------------------------------+   |
|                                                    |
+--------------------------------------------------+
|                                                    |
|  +--------------------------------------------+   |
|  |  [|||||||||||||||||||||||        ] 진행률    |   |
|  |   0        50       100 km/h               |   |
|  +--------------------------------------------+   |
|                                                    |
+--------------------------------------------------+
```

**설계 포인트:**
- 전체화면 모드 (상태바 숨김) - 운전 중 집중
- 속도 숫자 120sp, 화면 중앙 최대 크기
- 경과 시간은 속도 바로 아래
- 구간 달성 표시는 하단에 실시간 업데이트
- 진행률 바로 목표 속도까지 얼마나 남았는지 직관적 표시
- 가로 모드(Landscape) 대응 필수 - 차량 거치 시

### 5.1 기술 스택

### 5.3 궤적 렌더링 상세

| 60-100 km/h | 초록 | #4CAF50 | 중속 가속 |
| 100-150 km/h | 노랑 | #FFEB3B | 고속 가속 |

#### 렌더링 구현 전략

```
[구현 방식: 세그먼트 분할 방식]

GPS 포인트 배열: P0 -- P1 -- P2 -- P3 -- P4 -- ... -- Pn

각 세그먼트 (Pi -> Pi+1) 에 대해:
  - 평균 속도 = (speed[i] + speed[i+1]) / 2
  - 색상 = speedToColor(평균 속도)
  - osmdroid PolychromaticPaintList의 ColorMapping 구현

ColorMapping 구현:
  class SpeedColorMapping(
      private val speeds: List<Float>
  ) : ColorMapping {
      override fun getColorForIndex(index: Int): Int {
          val avgSpeed = (speeds[index] + speeds[index + 1]) / 2f
          return interpolateColor(avgSpeed)
      }
  }

색상 보간(interpolation):
  - 구간 경계에서 급격한 색 변화 방지
  - 인접 색상 간 선형 보간 사용
  - PolychromaticPaintList의 gradient=true 옵션 활용
```

| 속성 | 값 | 비고 |
|------|-----|------|
| 선 두께 | 8dp (외곽) + 5dp (내부) | 이중선으로 선명도 확보 |
| 외곽선 | #000000 (alpha 80%) | 지도 위에서 분리감 |
| 내부선 | 속도별 색상 | 그라데이션 활성화 |

| 안티앨리어싱 | true | 필수 |

#### 마커 표시 로직

```
마커 생성 조건:
  - 속도가 목표값을 처음으로 넘는 GPS 포인트 감지
  - 이전 포인트와 보간하여 정확한 도달 위치 추정
    예: P[i].speed = 98, P[i+1].speed = 103
        ratio = (100 - 98) / (103 - 98) = 0.4
        marker_lat = P[i].lat + 0.4 * (P[i+1].lat - P[i].lat)
        marker_lon = P[i].lon + 0.4 * (P[i+1].lon - P[i].lon)
        marker_time = 보간된 시간

마커 충돌 방지:
  - 마커 간 최소 거리: 30dp (화면 기준)
  - 겹칠 경우 줌 레벨에 따라 일부 숨김
  - 탭하면 말풍선(InfoWindow)으로 상세 표시
```

### 5.5 줌 & 바운딩 전략

```
자동 줌 로직:

1. 전체 궤적의 BoundingBox 계산
   bbox = BoundingBox(
       maxLat + padding,
       maxLon + padding,
       minLat - padding,
       minLon - padding
   )

2. 패딩: 상하좌우 15% 여유
   - 마커 레이블이 잘리지 않도록

3. 줌 레벨 제한
   - 최소: 14 (너무 멀어지지 않게)
   - 최대: 18 (너무 가까워지지 않게)

4. 지도 영역 크기에 따른 조정
   - 300dp 높이 영역: 세로 모드 기본
   - 전체화면: 지도 탭 시 확대 가능

5. 사용자 제스처
   - 핀치 줌: 활성
   - 드래그: 활성
   - 더블탭: 줌인
   - [리셋] 버튼: 전체 궤적 뷰로 복귀
```

### 5.6 지도 화면 배치 옵션 비교

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. 탭 전환 (권장) | [지도] [그래프] 탭 | 각각 넓은 공간, 집중 | 동시 비교 불가 |
| B. 상하 분할 | 지도 50% + 그래프 50% | 동시 확인 | 각각 너무 작음 |
| C. 전체화면 오버레이 | 지도 탭하면 전체화면 | 몰입감 | 전환 번거로움 |
| D. 스와이프 페이지 | 좌우 스와이프로 전환 | 직관적 | 탭이 더 명확 |

> 권장: A안 (탭 전환) 기본 + 지도 탭 시 전체화면 확대 옵션 (A+C 하이브리드)
> 이유: 결과 화면에서 Split Times가 가장 중요하고, 지도/그래프는 상세 확인용.
> 300dp 높이면 궤적 확인에 충분하고, 전체화면 확대로 상세 탐색 가능.

## 6. 데이터 구조 & 저장 전략

### 6.1 Room DB 스키마

#### 옵션 비교

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. 별도 테이블 (권장) | MeasurementRun + TrackPoint 1:N | 쿼리 유연, 인덱싱 가능 | 조인 필요 |
| B. JSON Blob | run 테이블에 points를 JSON으로 | 단순 저장 | 개별 포인트 쿼리 불가, 파싱 비용 |
| C. Protobuf Blob | 바이너리로 압축 저장 | 최소 용량 | 디버깅 어려움, 오버엔지니어링 |

> 권장: A안 (별도 테이블). 이유: 특정 구간 쿼리, 통계 집계, 향후 비교 기능에 필수.

#### 테이블 설계

```sql
-- 측정 세션 테이블
CREATE TABLE measurement_run (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at      INTEGER NOT NULL,       -- Unix timestamp (ms)
    mode            TEXT NOT NULL,           -- "0-100", "0-200", "custom" 등
    target_speed    REAL NOT NULL,           -- 목표 속도 (km/h)
    result_time     REAL,                    -- 결과 시간 (초, 소수점 2자리)
    max_speed       REAL,                    -- 최고 속도 (km/h)
    total_distance  REAL,                    -- 총 거리 (m)
    avg_accel_g     REAL,                    -- 평균 가속도 (G)
    vehicle_name    TEXT,                    -- 차량명
    memo            TEXT,                    -- 메모
    gps_accuracy    REAL,                    -- 평균 GPS 정확도 (m)
    gps_hz          INTEGER,                 -- GPS 수신율
    status          TEXT DEFAULT 'completed' -- completed, aborted, invalid
);

-- GPS 트랙포인트 테이블
CREATE TABLE track_point (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          INTEGER NOT NULL,        -- FK -> measurement_run.id
    seq             INTEGER NOT NULL,        -- 시퀀스 번호 (0, 1, 2, ...)
    timestamp_ms    INTEGER NOT NULL,        -- 기록 시점 (Unix ms)
    elapsed_ms      INTEGER NOT NULL,        -- 시작 후 경과 (ms)
    latitude        REAL NOT NULL,           -- 위도
    longitude       REAL NOT NULL,           -- 경도
    speed_kmh       REAL NOT NULL,           -- 속도 (km/h)
    accuracy_m      REAL,                    -- GPS 정확도 (m)
    altitude_m      REAL,                    -- 고도 (m)
    bearing         REAL,                    -- 방위각
    
    FOREIGN KEY (run_id) REFERENCES measurement_run(id) ON DELETE CASCADE
);

-- 구간 기록 테이블
CREATE TABLE split_time (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          INTEGER NOT NULL,        -- FK -> measurement_run.id
    speed_kmh       REAL NOT NULL,           -- 구간 속도 (30, 60, 80, 100, ...)
    elapsed_sec     REAL NOT NULL,           -- 도달 시간 (초)
    latitude        REAL,                    -- 도달 위치
    longitude       REAL,                    -- 도달 위치
    
    FOREIGN KEY (run_id) REFERENCES measurement_run(id) ON DELETE CASCADE
);

-- 인덱스
CREATE INDEX idx_track_point_run_id ON track_point(run_id);
CREATE INDEX idx_track_point_run_seq ON track_point(run_id, seq);
CREATE INDEX idx_split_time_run_id ON split_time(run_id);
CREATE INDEX idx_measurement_run_created ON measurement_run(created_at DESC);
CREATE INDEX idx_measurement_run_mode ON measurement_run(mode);
```

### 6.2 저장/렌더링 최적화

#### 용량 추정

| 항목 | 값 | 비고 |
|------|-----|------|
| 25Hz, 10초 측정 | 250 포인트 | 일반적 0-100 시나리오 |
| 25Hz, 30초 측정 | 750 포인트 | 0-200 시나리오 |
| TrackPoint 1개 크기 | ~100 bytes | 8개 필드 |
| 250 포인트 총 용량 | ~25 KB | 무시할 수 있는 수준 |
| 1000회 측정 누적 | ~25 MB | 전혀 문제 없음 |

#### 최적화 전략

```
[저장 최적화]

1. 배치 삽입 (Batch Insert)
   - 측정 중: 메모리 버퍼에 포인트 누적
   - 측정 완료 후: @Transaction으로 일괄 삽입
   - 이유: 개별 INSERT는 25Hz에서 IO 부하 유발

2. 다운샘플링 (지도 렌더링용)
   - 원본: 250개 포인트 전부 DB 저장
   - 렌더링: 줌 레벨에 따라 다운샘플
     - 줌 14-15: 매 5번째 포인트 (50개)
     - 줌 16-17: 매 2번째 포인트 (125개)
     - 줌 18+: 전체 포인트 (250개)
   - Douglas-Peucker 알고리즘 적용 가능 (향후)

3. 캐싱
   - 최근 10개 측정의 트랙포인트: 메모리 캐시
   - LRU 캐시 사이즈: 5MB (충분)

[렌더링 최적화]

4. Polyline 재사용
   - 결과 화면 진입 시 1회 생성, 탭 전환 시 재사용
   - 줌 변경 시 다운샘플 레벨만 변경

5. 지도 타일 캐싱
   - osmdroid 기본 캐시 활용 (SD 카드)
   - 최대 캐시: 100MB
```

### 8.3 과감한 가능성 (기획팀장 제안)

**"측정 영상 오버레이 기능"**

| 항목 | 내용 |
|------|------|
| 개요 | 측정 중 카메라 녹화 + 실시간 속도/시간 OSD 오버레이 |
| 벤치마크 | Dragy의 킬러 기능 - 영상 위에 속도계 표시 |
| 전략적 가치 | SNS 공유 시 폭발적 바이럴 효과, 경쟁앱 대비 최대 차별화 |
| 필요 리소스 | 추가 2-3주, CameraX + MediaCodec 지식 필요 |
| 리스크 | 구현 난이도 높음, 성능 이슈 가능 |
| 권장 | v2.0에는 미포함, v2.1 로드맵에 배치. 단, 지금부터 측정 데이터를 타임스탬프로 저장하여 나중에 영상과 동기화 가능하도록 준비 |

> "지금 안 하면 나중에 비싸지는 것":
> 타임스탬프 기반 데이터 저장 구조를 지금 잡아두지 않으면,
> 나중에 영상 오버레이 기능 추가 시 DB 마이그레이션 + 데이터 구조 변경이 필요해진다.
> 현재 설계의 timestamp_ms 필드가 이미 이를 대비한 것이다.
